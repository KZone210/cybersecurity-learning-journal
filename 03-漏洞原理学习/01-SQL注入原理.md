# SQL注入原理

程序员在编写代码时没有对用户提交的数据进行严格过滤。黑客编写特定的SQL语句提交到后端代码进行结构拼接，造成敏感数据泄露。



## 常见SQL注入分类

### 数字型

提交纯数字数据，并且后端代码在把提交数据带到SQL语句执行时没有加引号。

```shell
如果提交数据 id=2 and 1=1 后查到符合条件的数据（数据库有符合条件的数据），提交 id=2 and 1=2 后查不到符合条件的数据（数据库没有符合条件的数据）。查询结果随着后期加的请求数据变化，说明后期增加的请求数据" and 1=1"和" and 1=2"被后端代码正常生成SQL语句执行
```



### 字符型

提交字符串数据，后端处理时按照字符串处理

### 搜索型

后端处理SQL语句时按照提交数据进行like模糊匹配

### XX型

后端处理SQL语句时在筛选条件外面加了一层括号或其他符号

```shell
select * from user where username=('vince');  #测试SQL注入点时记得加阔号
```



## SQL注入点测试

### 引号测试

提交数据一个单引号或一个双引号，如果出现服务端mysql报错，这个数据一定被带到数据库自动执行了，说明百分百有sql注入点。如果是在MySQL命令终端执行非成对出现的单引号或双引号，那么MySQL只会让你补齐引号，不会报错。

所以从这个地方泄露的信息有：数据库版本、提交数据被后端代码自动执行

### 筛选条件测试

如果所有条件时可以查到数据，并且不满足条件时查不到数据。说明提交的数据没有经过过滤就被带到数据库中执行。

```shell
or 1=1#
and 1=1
and 1=2
```



## SQL注入不同视角解释

### 工程师视角
